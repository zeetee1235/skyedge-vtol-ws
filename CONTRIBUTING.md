# Contributing Guide

이 문서는 이 저장소에 코드를 추가하거나 수정할 때 따라야 하는 작업 절차를 설명합니다.

대상 독자:

- Git을 거의 처음 써보는 팀원
- GitHub에서 Pull Request를 아직 만들어본 적이 없는 팀원
- 브랜치가 왜 필요한지 익숙하지 않은 팀장 / 리뷰어

이 문서는 "실제로 어떻게 작업해야 하는가"에 초점을 맞춰 자세히 설명합니다.

## 먼저 이해해야 하는 핵심 개념

### 1. 저장소(Repository)

이 프로젝트 전체 파일이 들어 있는 Git 관리 폴더입니다.

현재 여러분이 작업하는 저장소 폴더가 이 프로젝트 작업 공간입니다.

### 2. 브랜치(Branch)

브랜치는 같은 프로젝트 안에서 작업 내용을 분리하는 "작업 줄기"입니다.

쉽게 말하면:

- `main`은 가장 안정적인 결과물을 보관하는 줄기
- `develop`은 팀이 같이 모아서 개발하는 줄기
- `feature/...`는 개인이 기능 하나를 작업하는 줄기

입니다.

브랜치를 나누는 이유는 다음과 같습니다.

- 다른 사람 작업과 충돌을 줄이기 위해
- 실수로 안정 버전을 망가뜨리지 않기 위해
- 기능별로 검토하고 합치기 쉽게 만들기 위해

### 3. 커밋(Commit)

커밋은 "이 시점의 변경 내용을 저장"하는 것입니다.

예를 들어:

- YOLO 인식 결과 필터 추가
- README 수정
- Foxglove 설정 문서 보완

같은 변경을 하나의 기록으로 남길 수 있습니다.

### 4. 푸시(Push)

내 컴퓨터에서 만든 커밋을 GitHub 원격 저장소에 올리는 것입니다.

### 5. Pull Request(PR)

내 브랜치의 작업을 `develop` 같은 기준 브랜치에 합쳐달라고 요청하는 절차입니다.

팀원 리뷰와 확인은 보통 PR에서 진행합니다.

## 이 저장소의 브랜치 전략

이 저장소는 아래 구조를 기본으로 사용합니다.

```text
main          ← 안정 버전
  └── develop ← 팀 통합 개발 브랜치
        └── feature/<패키지명>-<기능명> ← 개인 작업 브랜치
```

의미는 다음과 같습니다.

- `main`
  - 배포 또는 안정 버전 보관용입니다.
  - 직접 작업하지 않습니다.
  - 직접 push 하지 않습니다.
- `develop`
  - 팀에서 일상적으로 모으는 기준 브랜치입니다.
  - 새 작업은 보통 여기서 시작합니다.
- `feature/...`
  - 개인 기능 개발용 브랜치입니다.
  - 작업 하나마다 새로 만드는 것을 권장합니다.

## 브랜치 보호 규칙

팀 규칙은 아래와 같습니다.

| 브랜치 | 직접 push | PR 필수 | 최소 1명 승인 | CI 빌드 통과 |
|--------|-----------|---------|---------------|--------------|
| `main` | 불가 | 필수 | 필수 | 필수 |
| `develop` | 불가 | 필수 | 필수 | 불필수 |

즉:

- `main` 은 PR, 승인, CI 통과가 모두 필요합니다.
- `develop` 도 직접 push 하지 않고 PR과 승인 절차를 거칩니다.
- `develop` 은 CI 빌드 통과가 권장되지만 필수 규칙은 아닙니다.

## 가장 중요한 원칙

- 작업은 `develop`에서 시작해서 `feature/...` 브랜치에서 진행합니다.
- `main`에는 직접 push 하지 않습니다.
- `develop`에도 직접 push 하기보다 PR을 통해 반영하는 것을 기본으로 합니다.
- 작업 전에는 현재 브랜치가 무엇인지 항상 확인합니다.
- 남의 작업이 이미 올라와 있을 수 있으니 새 브랜치를 만들기 전 `develop`을 최신 상태로 맞춥니다.

## 작업 시작 전 확인

터미널에서 현재 브랜치를 확인합니다.

```bash
git branch --show-current
```

현재 변경 사항이 있는지도 확인합니다.

```bash
git status
```

`working tree clean` 이라고 나오면 현재 수정 중인 파일이 없다는 뜻입니다.

## 처음부터 작업 브랜치 만드는 방법

아래 순서를 그대로 따라 하면 됩니다.

### 1. 저장소 폴더로 이동

```bash
cd <저장소폴더>
```

예를 들어 저장소를 `~/skyedge-vtol-ws` 에 clone 했다면 아래처럼 이동합니다.

```bash
cd ~/skyedge-vtol-ws
```

### 2. `develop` 브랜치로 이동

```bash
git switch develop
```

만약 `switch`가 익숙하지 않거나 안 되면 아래 명령도 비슷한 역할입니다.

```bash
git checkout develop
```

### 3. 원격 최신 내용 가져오기

```bash
git pull
```

이 명령은 GitHub의 최신 변경을 내 컴퓨터 `develop` 브랜치에 반영합니다.

### 4. 새 작업 브랜치 만들기

형식:

```bash
git switch -c feature/<패키지명>-<기능명>
```

예시:

```bash
git switch -c feature/vtol-vision-yolo-confidence-filter
```

이 명령의 의미:

- `feature/...` 라는 새 브랜치를 만든다
- 만들자마자 그 브랜치로 이동한다

### 5. 브랜치가 잘 만들어졌는지 확인

```bash
git branch --show-current
```

예상 결과:

```text
feature/vtol-vision-yolo-confidence-filter
```

## 브랜치 이름 규칙

가능하면 아래 형식을 권장합니다.

```text
feature/<패키지명>-<기능명>
fix/<패키지명>-<문제명>
docs/<문서명>
refactor/<대상>-<설명>
```

예시:

```text
feature/vtol-control-nav-waypoint-reset
fix/vtol-vision-aruco-missing-frame
docs/git-guide
refactor/bringup-launch-cleanup
```

브랜치 이름 작성 팁:

- 띄어쓰기는 쓰지 않습니다.
- 한글보다 영문 소문자와 하이픈 `-` 사용을 권장합니다.
- 너무 길지 않게, 하지만 의미는 드러나게 적습니다.

## 코드 수정 후 해야 하는 일

작업을 마친 뒤에는 아래 순서를 따릅니다.

### 1. 어떤 파일이 바뀌었는지 확인

```bash
git status
```

### 2. 변경 내용을 확인

```bash
git diff
```

이 명령은 무엇이 바뀌었는지 보여줍니다.

### 3. 커밋에 담을 파일 선택

특정 파일만 추가:

```bash
git add README.md
git add CONTRIBUTING.md
```

현재 변경된 파일 전체 추가:

```bash
git add .
```

처음에는 `git add .` 가 편하지만, 익숙해지면 필요한 파일만 선택하는 습관이 더 좋습니다.

### 4. 다시 상태 확인

```bash
git status
```

`Changes to be committed` 항목에 올라가 있으면 커밋 준비가 된 것입니다.

### 5. 커밋 만들기

```bash
git commit -m "docs: 초보자용 Git 작업 가이드 추가"
```

커밋 메시지는 짧고 분명하게 작성합니다.

좋은 예:

```text
feat: yolo 노드 탐지 신뢰도 임계값 파라미터 추가
fix: aruco 노드 빈 카메라 프레임 처리
docs: 초보자용 기여 가이드 보강
```

피하면 좋은 예:

```text
fix
update
final
수정
```

## 원격 저장소에 올리기

브랜치를 처음 올릴 때는 아래 명령을 사용합니다.

```bash
git push -u origin feature/vtol-vision-yolo-confidence-filter
```

의미:

- `origin`: GitHub 원격 저장소 이름
- `feature/...`: 내가 만든 작업 브랜치 이름
- `-u`: 다음부터는 `git push` 만 입력해도 되도록 연결

브랜치를 이미 한 번 올린 뒤에는 아래처럼 간단히 해도 됩니다.

```bash
git push
```

## Pull Request 만드는 방법

브랜치를 GitHub에 올린 뒤에는 PR을 만듭니다.

기본 대상은 `develop` 입니다.

### GitHub 웹에서 만드는 순서

1. GitHub 저장소 페이지로 이동합니다.
2. 방금 push한 브랜치에 대해 `Compare & pull request` 버튼이 보이면 클릭합니다.
3. 대상 브랜치가 `develop` 인지 확인합니다.
4. 제목을 입력합니다.
5. 본문에 변경 내용과 테스트 방법을 적습니다.
6. 리뷰어가 있다면 지정합니다.
7. PR을 생성합니다.

### PR 제목 예시

```text
[vtol_vision_yolo] 탐지 신뢰도 임계값 파라미터 추가
[docs] 초보자용 기여 가이드 보강
```

관련 이슈가 있다면 PR 본문에 아래처럼 연결하는 것을 권장합니다.

```text
Closes #12
관련 이슈 #18
```

### PR 본문 예시

```text
## 변경 내용
- YOLO 탐지 신뢰도 임계값 파라미터 추가
- 기본값과 로그 메시지 정리

## 테스트
- colcon build
- 노드 실행 후 탐지 결과 확인

## 참고 사항
- 기존 토픽 이름은 변경하지 않음
```

## GitHub Issue 사용

문제가 생기거나, 기능 제안이 있거나, 나중에 해야 할 일을 기록해야 한다면 GitHub Issue를 사용하는 것을 권장합니다.

이슈는 아래 상황에서 특히 유용합니다.

- 버그 발생
- 기능 제안
- 문서 보완 요청
- 나중에 할 일 기록
- 팀 논의 필요

초보자용 자세한 설명은 아래 문서를 참고하세요.

- [docs/github_issues_guide.md](docs/github_issues_guide.md)

## GitHub Actions 자동 검사

이 저장소에서는 PR 또는 주요 브랜치 push 시 자동 검사가 실행됩니다.

현재 자동으로 확인하는 항목:

- ROS 2 워크스페이스 빌드 가능 여부
- Docker 이미지 빌드 가능 여부
- 기본 Python 테스트 스위트 통과 여부

즉, 리뷰어가 코드를 직접 받아서 바로 빌드해보지 않더라도 기본적인 빌드 실패는 미리 발견할 수 있습니다.

PR을 올렸다면 GitHub 화면에서 Actions 또는 체크 상태를 같이 확인하는 습관을 권장합니다.

초보자를 위한 운영 원칙:

- `main`, `develop` 는 가능한 한 항상 초록 상태를 유지합니다.
- 아직 구현하지 않은 기능 테스트는 이유를 적어서 `skip` 처리합니다.
- 기능 구현을 시작한 담당자는 해당 테스트의 `skip` 을 제거하고 구현과 함께 PASS 상태로 만듭니다.

현재 브랜치 확인:

```bash
git branch --show-current
```

상태 확인:

```bash
git status
```

변경 내용 확인:

```bash
git diff
```

원격 브랜치 포함 전체 브랜치 보기:

```bash
git branch -a
```

최근 커밋 확인:

```bash
git log --oneline -5
```

## 실수하기 쉬운 상황과 대처법

### 1. `main`에서 바로 작업해버린 경우

아직 커밋 전이라면:

- 지금 수정 내용을 유지한 채 `feature/...` 브랜치를 새로 만들면 됩니다.

```bash
git switch -c feature/<패키지명>-<기능명>
```

아직 커밋하지 않았더라도 변경 내용은 그대로 따라옵니다.

### 2. `develop`에서 바로 커밋해버린 경우

혼자 해결하기보다 팀에 먼저 알려주는 것이 안전합니다.

상황에 따라 커밋을 새 브랜치로 옮길 수 있지만, 초보자에게는 잘못된 복구가 더 위험할 수 있습니다.

### 3. `git pull` 했더니 충돌이 난 경우

무리해서 바로 해결하지 말고 팀원이나 담당자와 같이 확인하는 것을 권장합니다.

충돌은 두 사람이 같은 부분을 다르게 수정했을 때 주로 발생합니다.

### 4. 다른 사람 브랜치에 push 해버릴까 걱정되는 경우

아래 명령으로 현재 브랜치를 먼저 확인합니다.

```bash
git branch --show-current
```

이 습관 하나만으로도 실수를 크게 줄일 수 있습니다.

## 리뷰어 / 팀장을 위한 최소 확인 포인트

Git을 깊게 몰라도 아래 항목만 보면 기본적인 확인은 가능합니다.

- PR 대상 브랜치가 `develop` 인가
- 제목과 본문에 변경 목적이 적혀 있는가
- 어떤 파일이 바뀌었는지 설명이 있는가
- 테스트 방법이 적혀 있는가
- 안정 버전(`main`)에 직접 반영하려는 내용은 아닌가

## 문서 수정 시 원칙

- 새로운 실행 방법이 생기면 README 또는 관련 문서를 같이 수정합니다.
- 설치 방법이 바뀌면 반드시 초보자 기준으로 다시 읽어봅니다.
- 명령어 예시는 복사해서 바로 실행할 수 있게 적습니다.
- "알아서", "적당히", "필요하면" 같은 표현은 최대한 줄입니다.

## 추천 작업 순서 요약

초보자는 아래 순서를 거의 그대로 사용하면 됩니다.

```bash
git switch develop
git pull
git switch -c feature/<패키지명>-<기능명>
# 작업
git status
git add .
git commit -m "feat: 설명"
git push -u origin feature/<패키지명>-<기능명>
```

그다음 GitHub에서 `develop` 대상으로 PR을 생성합니다.
